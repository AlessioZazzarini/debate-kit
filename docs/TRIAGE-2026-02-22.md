# Debate Skill Triage: GPT-5.2 Cross-Model Review

**Date:** 2026-02-22
**Reviewer:** Codex CLI (GPT-5.2, nicknamed "Tony")
**Target:** `.claude/skills/debate/SKILL.md`
**Mode:** `/debate review .claude/skills/debate/SKILL.md`
**Findings:** 11 issues, 4 marked critical

## Root Cause: Prompt Bias

The code review prompt (Step 4c) listed **Security** as priority #1 with the most specific language. For a local dev tool (a markdown skill file running in the developer's own terminal), this is the wrong priority. The "attacker" would be the developer themselves. There's no public API, no untrusted input, no multi-tenant access.

**Result:** 4 of 11 findings were security-focused. 3 of 4 "criticals" were security theater.

## Full Triage

### Dismissed (security theater for a local dev tool)

| # | Finding | Severity (Tony) | Why Dismissed |
|---|---------|-----------------|---------------|
| 1 | Path prefix bypass (`/repo-malicious` matches `/repo`) | Critical | The developer types the path themselves. They're not attacking their own tool. The `realpath` + prefix check is defense-in-depth, not a production auth boundary. |
| 2 | TOCTOU race on validated path | Critical | Who's swapping symlinks on your laptop between two shell commands? This is a theoretical concern for a single-user CLI tool. |
| 6 | Terminal escape injection via stderr | Critical | Stderr comes from Claude CLI / Codex CLI -- tools the developer installed and trusts. Not untrusted input. |
| 9 | No `.env` guardrails for reviewer | Medium | The reviewer reads YOUR repo with YOUR permission. Adding a denylist prevents legitimate code review of files near `.env`. The model providers already have data policies. |

### Accepted (real usability/correctness bugs)

| # | Finding | Severity (Tony) | Root Cause | Fix Applied |
|---|---------|-----------------|------------|-------------|
| 5 | `EXIT_CODE=$?` unreliable across separate Bash tool calls | High | Each Bash tool invocation is a separate shell in Claude Code. `$?` in a subsequent call is always 0. | **Fix C:** Chain `; echo "EXIT_CODE:$?"` in the same Bash call as the invocation. Parse from tool output. |
| 7 | Claude session resume picks `.[0].id` -- may be wrong session | High | `claude sessions list --json` ordering isn't guaranteed. Could resume a different session. | **Fix B:** Dropped session resume entirely. Fresh invocations with prior feedback in context pack for all rounds. |
| 8 | Codex session ID extraction assumes stderr is JSON | High | Codex stderr is human-readable text, not JSON. The `jq` parse would always fail. | **Fix B:** Same fix -- dropped `codex resume` entirely. |
| 10 | Verdict parsing underspecified for critical count | Medium | Pattern `VERDICT:\s*(\d+)\s+issues?\s+found` doesn't capture critical count separately. | **Fix D:** Standardized to `VERDICT: issues=N critical=N` across all three modes. Unambiguous, machine-parseable. |
| 11 | Cleanup not enforced (no `trap`) | Low | Can't use `trap` in Claude Code's tool model (no persistent shell). | Acknowledged in orchestrator lessons. The skill instructions already specify cleanup in all code paths (Step 7, Step 5.1, Step 5.2). |

### Deferred (not worth fixing)

| # | Finding | Severity (Tony) | Why Deferred |
|---|---------|-----------------|--------------|
| 3 | `realpath --` not portable on macOS | Low | `realpath` works on modern macOS (12+). The `--` is just defensive end-of-options. Low risk. |
| 4 | `mktemp -d /tmp/debate.XXXXXX` portability | Low | We literally ran it on macOS during testing and it worked. False positive. |

## Fixes Applied

### Fix A: Rebalanced Code Review Prompt (Step 4c)

**Before:** Security #1, Correctness #5.
**After:** Correctness #1, Security #5 with "(for production-facing code)" qualifier.

Added anti-theater directive:
> Prioritize issues that would cause real failures over theoretical concerns.
> Do NOT flag issues that only matter in adversarial/multi-tenant contexts
> unless the code actually runs in such an environment.

### Fix B: Dropped Session Resume (Both Providers)

**Before:** Round 2+ used `--resume $SESSION_ID` (Claude) or `codex resume` (Codex) with fragile ID extraction.
**After:** Fresh invocations for all rounds. Context pack rebuilt with revised plan + prior review feedback.

Why:
- Claude CLI session ID extraction (`.[0].id`) has no guaranteed ordering
- Codex CLI stderr is human text, not JSON -- `jq` parse always fails
- Fresh invocations cost the same (`--max-budget-usd 0.50` per round)
- Simpler, more reliable, no cross-provider inconsistency

### Fix C: Exit Code in Same Bash Call (Step 5.1)

**Before:** `EXIT_CODE=$?` in a separate Bash tool call (always returns 0).
**After:** `; echo "EXIT_CODE:$?"` appended to the invocation command. Parse from tool output.

### Fix D: Standardized Verdict Format (All Modes)

**Before:**
- Plan mode: `VERDICT: REVISE -- [number] issues found, [number] critical`
- Debug mode: `VERDICT: REVISE -- diagnosis has [number] gaps`
- Review mode: `VERDICT: [number] issues found, [number] critical`

**After (all modes):**
- `VERDICT: APPROVED`
- `VERDICT: REVISE issues=N critical=N`
- `VERDICT: issues=N critical=N` (review mode, no APPROVED/REVISE prefix)

Machine-parseable regex: `issues\s*=\s*(\d+)\s+critical\s*=\s*(\d+)`

## Lessons Learned

1. **Prompt priority ordering biases the reviewer.** Whatever category is listed first with the most specific language gets the most findings. For dev tools, security should be deprioritized.
2. **Session resume across CLI tools is unreliable.** Neither Claude CLI nor Codex CLI provides a reliable programmatic way to resume sessions from a parent orchestrator. Fresh invocations with context packing are simpler and equally effective.
3. **Cross-model review works.** Tony found 4 real bugs that the original author missed. The security theater findings were a prompt calibration issue, not a review quality issue.
4. **Triage the reviewer, don't just accept everything.** AI reviewers are susceptible to prompt bias just like human reviewers are susceptible to checklist bias. The value is in the triage, not the raw findings.
